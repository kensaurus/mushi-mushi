## The evolution loop for AI-assisted software

The vibe-coding era solved the creation bottleneck. AI agents build and ship in hours. The new bottleneck is **selection pressure**: out of everything you could fix next, which changes actually make the app better for real users?

Without a closed loop: users churn silently, agents write code without knowing which bugs keep reappearing, every sprint starts where the last one ended.

With the evolution loop: every user friction event becomes structured signal, every merged fix promotes a lesson rule into the codebase, every future agent run starts with accumulated knowledge.

Mushi turns the random walk into **cumulative selection**. Every user report is a selection event — captured with context, triaged by AI, fixed by an agent, scored by a judge, and encoded as a permanent lesson. Every iteration starts from a fitter baseline than the last. [Read the five-stage walkthrough &rarr;](/concepts/evolution-loop)

This is **selection with memory** — [Dawkins&rsquo; phrase](https://en.wikipedia.org/wiki/Climbing_Mount_Improbable) for why cumulative selection beats random search by orders of magnitude.

## Four properties selection requires

Sentry sees what your code throws. Mushi sees what your users feel. The two
are designed to compose: Mushi&rsquo;s classifier reads Sentry breadcrumbs,
and Sentry User Feedback events forward into Mushi via webhook &mdash; so
nothing about your existing setup has to change. But Mushi adds what selection
needs and Sentry alone cannot provide:

- **Variation capture**: every report is embedded and clustered. The same broken button collapses to one row, not 200 duplicate tickets.
- **Selection pressure**: when a cluster reaches coherence &ge; 0.75 and size &ge; 3, it is promoted to a named lesson and synced to `.mushi/lessons.json` in your repo.
- **Inheritance**: lessons are ranked by severity and packed to &le;&thinsp;3&thinsp;000 tokens before being injected into PR review context &mdash; so neither human reviewer nor AI agent can repeat the same class of mistake.
- **Reward loop**: the user who found the mutation is credited in the changelog. The SDK shows them a toast: *&ldquo;v1.2.3 shipped &mdash; you helped fix this.&rdquo;* This closes the feedback loop that keeps beta testers reporting.
