## Know why it broke — then close the loop

The vibe-coding era solved the *creation* bottleneck: AI ships in hours. The new bottleneck is **selection** — out of everything you could fix next, which change actually makes the app better for real users? Mushi turns each user-felt bug into a plain-English diagnosis and a paste-ready fix, then encodes every merged fix as a lesson so the next agent run starts smarter.

That is **selection with memory**: cumulative selection beats random search by orders of magnitude. The longer thesis (Dawkins, Taleb, the NTSB black box) lives in the docs. [Read the thesis &rarr;](/concepts/closed-loop) &middot; [The five-stage loop &rarr;](/concepts/evolution-loop)

## Four properties selection requires

Sentry sees what your code throws. Mushi sees what your users feel. The two
are designed to compose: Mushi&rsquo;s classifier reads Sentry breadcrumbs,
and Sentry User Feedback events forward into Mushi via webhook &mdash; so
nothing about your existing setup has to change. But Mushi adds what selection
needs and Sentry alone cannot provide:

- **Variation capture**: every report is embedded and clustered. The same broken button collapses to one row, not 200 duplicate tickets.
- **Selection pressure**: when a cluster reaches coherence &ge; 0.75 and size &ge; 3, it is promoted to a named lesson and synced to `.mushi/lessons.json` in your repo.
- **Inheritance**: lessons are ranked by severity and packed to &le;&thinsp;3&thinsp;000 tokens before being injected into PR review context &mdash; so neither human reviewer nor AI agent can repeat the same class of mistake.
- **Reward loop**: the user who found the mutation is credited in the changelog. The SDK shows them a toast: *&ldquo;v1.2.3 shipped &mdash; you helped fix this.&rdquo;* This closes the feedback loop that keeps beta testers reporting.
