## The old loop and the new loop

Software development has followed one model for 50 years: PM defines spec → dev implements → QA tests → ships → user encounters a bug → support ticket → someday fixed. The feedback loop takes weeks; the lesson is lost.

Mushi flips this. Every user report is embedded, clustered with similar ones, and promoted to a **named learning rule** — then injected into the next PR review and the next agent fix run. The user who found the bug gets credited in the changelog. The next developer can&rsquo;t repeat the same mistake because the rule is already in the review context.

This is **cumulative selection with memory** — [Dawkins&rsquo; phrase](https://en.wikipedia.org/wiki/Climbing_Mount_Improbable) for why selection beats random search. Every iteration starts from a better baseline than the last.

## Software&rsquo;s NTSB moment

Sentry sees what your code throws. Mushi sees what your users feel. The two
are designed to compose: Mushi&rsquo;s classifier reads Sentry breadcrumbs,
and Sentry User Feedback events forward into Mushi via webhook &mdash; so
nothing about your existing setup has to change. But Mushi goes further:

- **Mistake DB**: every report is embedded and clustered. The same broken button collapses to one row, not 200 duplicate tickets.
- **Learning rules**: when a cluster reaches coherence ≥ 0.75 and size ≥ 3, it is promoted to a named lesson and synced to `.mushi/lessons.json` in your repo.
- **Token-budget injection**: lessons are ranked, reranked by severity, and packed to ≤ 3 000 tokens before being injected into PR review context — so neither human reviewer nor AI agent can repeat the same class of mistake.
- **Reporter attribution**: the user who found the bug is credited in the changelog. The SDK shows them a toast: *&ldquo;v1.2.3 shipped — you helped fix this.&rdquo;* This closes the feedback loop that makes beta testers keep reporting.
