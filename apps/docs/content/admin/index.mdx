# Admin console

The admin console is the command surface for everything Mushi does after a report
arrives. It is a React SPA at `app.mushimushi.dev` (or your self-hosted equivalent).

All 30+ pages are organised around the **Plan → Do → Check → Act** loop. The sidebar
groups them accordingly. Use the Command Palette (`⌘K` / `Ctrl K`) to jump to any page,
report, or fix attempt by name.

## Navigation model

The sidebar is always visible on desktop. Sections collapse/expand per-project.

The top header shows:
- **Project switcher** — jump between projects without leaving the current surface.
- **Org switcher** — for users in multiple organisations.
- **Presence badge** — how many teammates are in the console right now.
- **Notifications bell** — fix completions, judge drift alerts, tier-up events.
- **What's New** — annotated changelog with per-feature badges.

<Callout type="info">
**Command Palette** (`⌘K` / `Ctrl K`) reaches any page, report, fix attempt, or graph node
by name, and exposes quick actions (dispatch a fix, trigger a gate run, export a dataset)
without leaving the current page.
</Callout>

## First run — onboarding checklist

When you create a project, the **Setup checklist** appears on the `/onboarding` page:

1. **Add SDK** — install `@mushi-mushi/web` or `@mushi-mushi/react-native` and provide your project API key.
2. **First report** — send a test report via the SDK or the **Send test report** button on the Projects page.
3. **Connect integrations** — wire up BYOK keys on the [Settings → Integrations health](/admin/health) page.
4. **First fix** — dispatch an approved triage decision from the Reports page.
5. **Invite a teammate** — optional, via Organization members.

The checklist persists in the header until all required steps are complete. If the SDK heartbeat doesn't arrive within an hour, a diagnostics card shows exactly which call is failing.

## Self-hosting

The admin console is a static SPA deployed from `apps/admin/`. See [Self-hosting → Admin SPA](/self-hosting/admin-spa-deploy).

It reads `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` at build time. All other
configuration flows through project settings stored in your Supabase instance.
