## v1.1.x

Latest tag: `v1.1.0`

### Highlights — Rewards program

The Rewards program is the biggest new surface since v1.0.0. It turns passive
reporters into engaged contributors by awarding points for in-app activity
and bug reports, organising users into tiers, and unlocking configurable perks.

- **`identify(userId, traits?)`** — new SDK method on all platforms (Web,
  React, React Native). Links the current user to their Mushi `end_users`
  record so reports, activity, and points all attribute correctly. Call it on
  auth state change; the SDK queues any events fired before the first
  `identify()` and back-fills them on flush.

- **`submitActivity(events[])`** — batch-submits custom activity events to
  `POST /v1/sdk/activity`. Use for domain-specific actions your app cares
  about (e.g. `lesson_complete`, `video_watched`).

- **`getReputation()` / `getTier()`** — query the current user's point totals
  and tier. Available on the SDK instance and as `useReputation()` / `useTier()`
  React hooks.

- **`<MushiRewardsBadge />`** (`@mushi-mushi/react`) — polymorphic drop-in
  badge that renders the user's tier name and (optionally) their point count.

- **`MushiRewardsConfig`** — new `rewards` key in init config. Options:
  `enabled`, `trackActivity`, `consentMode` (`explicit` / `auto`),
  `showInWidget`, `showNotifications`, `flushIntervalMs`, `verifyUserToken`.

- **Auto-activity tracking** (`trackActivity: true`) — the Web SDK
  automatically captures `session_start`, `page_view`, `navigate`, and
  `button_press` events without any code changes. The React Native SDK captures
  screen changes via `setScreen()`.

- **Admin console: Rewards page** — new `/rewards` section with 8 tabs:
  Overview, Activity Rules, Tier Ladder, Contributors, Quests, Retention,
  Simulator, and Settings.

- **Tier Ladder** — four built-in tiers (Free → Explorer → Contributor →
  Champion). Fully customisable name, threshold, perks, and point multiplier.
  Tier-up fires a configurable webhook automatically.

- **Quests** — multi-step goals. Define a sequence of actions and a time
  window; the server evaluates completion after each activity flush.

- **Webhooks** — `points_awarded`, `tier_up`, `quest_complete` events delivered
  to any HTTPS endpoint. HMAC-SHA256 signed. Test-fireable from the admin UI.

- **GDPR** — `GET /v1/sdk/me/export` (data portability), `DELETE /v1/sdk/me`
  (erasure), `POST /v1/sdk/me/consent` (withdraw / re-grant). Consent mode
  `explicit` shows a one-time prompt in the SDK widget.

- **Anti-gaming composability** — the existing anonymous `reporter_token_hash`
  reputation gate applies to rewards. Known bad actors earn 0 points on all
  activity events until their reputation recovers.

- **P2: JWT identity verification** — a `verifyUserToken` callback lets the
  host app supply a signed JWT so the server can verify the user's identity
  before issuing monetary-eligible awards.

- **P2: Monetary payouts** — Stripe Connect Express onboarding, per-user
  payout accounts, a monthly payout aggregator, and a liability dashboard.
  Available on the Enterprise plan.

- **P3: Anomaly detection** — nightly cron flags users whose activity pattern
  deviates statistically from the cohort median. Flagged records land in
  `reward_disputes` for manual review.

### Migration

No breaking changes. Rewards are **opt-in** — existing apps work without
modification. To enable:

1. Add `rewards: { enabled: true }` to your `init()` config.
2. Call `identify(userId, traits)` on auth state change.
3. Go to **Admin → Rewards → Settings** and flip the toggle.

## v1.0.x

Latest tag: `v1.0.0`

### Highlights

- **`createBreadcrumbBuffer(options?)`** — — 50-entry ring buffer of `{ timestamp, category, level, message, data? }`. Long messages truncate at 500 chars _at insert time_ so a runaway log line can't push useful crumbs out of the buffer. PII scrubbing runs at report-snapshot time (not at insert) so in-app debugging stays unredacted while the wire payload stays clean.
- **`normaliseThrown(thrown)`** — — turns any thrown value (`Error`, string, plain object, `null`, frozen `DOMException`) into `{ name, message, stack?, cause? }` with an 8 KB stack cap and cyclic-cause guards. Powers `Mushi.captureException()`; exposed so adapters (Vue / Svelte / Angular / RN) can ship their own thin wrappers without re-implementing the normalisation.
- **`captureEnvironment()` expansion** — — same one-call API, richer payload: viewport + screen + DPR, **User-Agent Client Hints** when supported (`brands`, `mobile`, `platform`, `model`, plus the high-entropy values resolved out-of-band and folded back in on the next capture), accessibility prefs (`prefers-reduced-motion`, `prefers-color-scheme`, `prefers-contrast`), online / displayMode / page title, and a one-shot **page-load timing** read from `PerformanceNavigationTiming` (TTFB, DOMContentLoaded, FCP, LCP). Every individual field stays optional so a Safari / Firefox / iOS WebView still produces a useful payload.
