### C1. Sentry DSN 403 spam â€” no circuit breaker

**Severity:** Critical (UX) â€” pollutes devtools network panel, wastes battery, hides real failures.
**Where:** `apps/admin/src/lib/sentry.ts`
**Symptom:** Every captured event POSTed to `o4510538320314368.ingest.us.sentry.io/...` returned 403 because the DSN was rotated/disabled. Sentry retried indefinitely.
**Root cause:** Default `makeFetchTransport` has no opinion about persistent auth failures â€” it keeps trying.
**Fix:** Wrapped the upstream transport in a self-disabling `makeCircuitBreakingTransport`. After 3 consecutive 401/403 responses the breaker trips and subsequent envelopes short-circuit to `{ statusCode: 200 }` with one warning log. 429 still flows through to upstream rate-limit logic, network errors don't trip the breaker (transient).

```62:98:apps/admin/src/lib/sentry.ts
function makeCircuitBreakingTransport(options: BrowserTransportOptions): Transport {
  const upstream = makeFetchTransport(options)
  // `createTransport` expects a `makeRequest(request) => Promise<Response>`
  // function, not an envelope; the upstream transport's `.send(envelope)`
  // wraps that internally. We just delegate to upstream.send so we observe
  // the same response shape and bookkeeping the SDK uses.
  return createTransport(options, async (_request): Promise<SendResult> => {
    if (transportDisabled) {
      return { statusCode: 200 } as SendResult
    }
    try {
      const response = await upstream.send(_request as unknown as SendArg)
      const status = response?.statusCode
      if (status === 401 || status === 403) {
        consecutiveAuthFails += 1
        if (consecutiveAuthFails >= AUTH_FAIL_THRESHOLD) {
          transportDisabled = true
          // ... single warning, then permanent no-op for this page lifetime
```

**Verification:** TypeScript build passes; types derived from public `makeFetchTransport` signature so the fix is resilient across Sentry minor versions.

### C2. N+1 query storm on every page load

**Severity:** Critical (perf) â€” `/v1/admin/setup` was being hit **12+ times in parallel** per render, doubled in StrictMode (24+ times) for a single page load.
**Where:** `apps/admin/src/lib/supabase.ts`
**Root cause:** ~17 components call `useSetupStatus()` (DashboardPage, FixesPage, NextBestAction, QuickstartMegaCta, FirstRunTour, SetupNudge, ProjectSwitcher, GettingStartedEmpty, â€¦). Each mounts independently; `usePageData` was firing a fresh fetch per consumer. Same pattern hit `/v1/admin/dashboard` and `/v1/admin/billing`.
**Fix:** Added in-flight request deduplication + 200ms TTL micro-cache in `apiFetch`:

- Concurrent identical GET/HEAD requests (same `method:path`, no body) share one promise.
- Resolved values are held for 200ms so a component mounting 50ms later doesn't refetch.
- POST/PATCH/DELETE are NEVER deduped (state mutations must always run).
- `cache: 'no-store'` opts out of dedup (used by `usePageData.reload()`).

// The admin console mounts ~17 components that each call `useSetupStatus()`
// (DashboardPage, FixesPage, NextBestAction, QuickstartMegaCta, FirstRunTour,
// SetupNudge, ProjectSwitcher, GettingStartedEmpty, etc). Without dedup,
// every page load fires GET /v1/admin/setup 12+ times in parallel â€” wasted
// bandwidth, wasted Supabase function invocations, and duplicated render-time
// loading flickers. Same problem with /v1/admin/dashboard and /v1/admin/billing.

### C3. API failures silently invisible in Sentry

**Severity:** Critical (observability) â€” only React render errors were reaching Sentry; HTTP 4xx/5xx and network failures stayed in `console.warn` only.
**Where:** `apps/admin/src/lib/supabase.ts`
**Fix:** All non-2xx responses now produce a Sentry breadcrumb (warning for 4xx, error for 5xx). 5xx escalates to `captureMessage` so server bugs surface even when the UI doesn't crash. Network errors (DNS, TLS, offline, CORS) become `captureException` â€” except `AbortError`, which is expected control-flow signal.

```135:185:apps/admin/src/lib/supabase.ts
    if (!res.ok) {
      const body = await res.text()
      debugWarn('api', `${method} ${path} â†’ ${res.status} (${ms}ms)`, { body: body.slice(0, 200) })
      // Telemetry: every non-2xx becomes a Sentry breadcrumb so we get a
      // request log for free in any Sentry event captured later in the same
      // session. 5xx is escalated to a captureMessage because server errors
      // are bugs we want to know about even when the user never crashes the
      // UI. 4xx (auth, validation) is intentionally NOT escalated to keep
      // the issue list signal-rich; auth flows trip 401 by design.
      Sentry.addBreadcrumb({
        category: 'api',
        type: 'http',
        level: res.status >= 500 ? 'error' : 'warning',
        data: { method, url: path, status_code: res.status, duration_ms: ms },
        message: `${method} ${path} â†’ ${res.status}`,
      })
      if (res.status >= 500) {
        Sentry.captureMessage(`API ${res.status} ${method} ${path}`, {
          level: 'error',
          tags: { source: 'apiFetch', http_status: String(res.status), api_path: path },
          contexts: { http: { method, url: path, status_code: res.status, duration_ms: ms } },
          extra: { response_snippet: body.slice(0, 500) },
        })
      }
```

**Why 4xx is breadcrumb-only:** auth flows trip 401 by design; validation flows trip 422 by design. Escalating those to `captureMessage` would drown the Sentry issue list in expected-control-flow noise.

### C5. TypeScript build broken by Sentry transport types

**Severity:** P0 (build) â€” `tsc -b` was failing with `TS2305` and `TS2307` after the C1 changes because `BrowserTransportOptions`, `Transport`, `Envelope`, and `TransportMakeRequestResponse` are not part of `@sentry/react`'s public re-export surface, and `@sentry/core` is not a direct dependency.
**Fix:** Derived all needed types from the public `makeFetchTransport` signature instead of importing internal types. This also makes the code resilient to Sentry minor-version reshuffles.

```33:44:apps/admin/src/lib/sentry.ts
import { makeFetchTransport, createTransport } from '@sentry/react'

// Sentry's transport-related types (Transport, Envelope, BrowserTransportOptions,
// TransportMakeRequestResponse) live inside @sentry/core and are NOT re-exported
// from @sentry/react's public surface (verified against @sentry/react@10.49.0
// and @sentry/browser@10.49.0). Rather than reach into an internal subpath
// that could move between minor versions, derive the few we need from the
// public function signatures.
type BrowserTransportOptions = Parameters<typeof makeFetchTransport>[0]
type Transport = ReturnType<typeof makeFetchTransport>
type SendArg = Parameters<Transport['send']>[0]
type SendResult = Awaited<ReturnType<Transport['send']>>
```
