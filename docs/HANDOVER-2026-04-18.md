## Architecture quick-ref

### Pipeline data flow

```
SDK report  ──POST /v1/reports (api function)
              │
              ▼  (fire-and-forget, EdgeRuntime.waitUntil)
            fast-filter  ─Haiku→  reports.{category,severity,confidence,summary,...}
              │                   llm_invocations row + langfuse_trace_id
              │
              ▼  (only if confidence < threshold OR explicit deep-classify)
            classify-report ─Sonnet→  same row, additional fields
                                       another llm_invocations row + langfuse_trace_id
              │
              ▼  (nightly cron OR /v1/admin/judge/run)
            judge-batch  ─GPT-4.1→  classification_evaluations rows
                                    cron_runs row
                                    each eval has langfuse_trace_id

Admin clicks "Dispatch fix"
              │
              ▼  POST /v1/admin/fixes/dispatch (api function)
              ▼  fire-and-forget invoke (verify_jwt=false)
            fix-worker  ─Sonnet 4.5 via OpenRouter→  
                          ├── RAG over project_repos embeddings
                          ├── generateObject with fixSchema (Zod)
                          ├── opens GitHub draft PR (REST)
                          └── fix_attempts row with langfuse_trace_id, pr_number, files
```

### Why some functions have `verify_jwt = false`

`api`, `fast-filter`, `classify-report`, `judge-batch`, `fix-worker` all sit behind `verify_jwt = false` in `supabase/config.toml`. After Supabase rotated to the `sb_secret_*` key format, the platform gateway started 401-rejecting every service-role inter-function call (`UNAUTHORIZED_INVALID_JWT_FORMAT`). These functions are internal-only and self-validate via body params (`reportId` / `projectId`) plus `jwtAuth` middleware in `api`. **If you add a new internal function, add it to that block too.**

Cloud deploys must use `--no-verify-jwt`:

```bash
cd packages/server
npx supabase functions deploy fast-filter --no-verify-jwt
```

## Wave E — Whitepaper gap closure + Firecrawl + self-improving loop (2026-04-18)

The "fill every Whitepaper V5 gap, add a self-improving loop, and bolt on optional web crawling" sweep. Six phases, all landed in this commit.

### Phase 1 — Firecrawl BYOK groundwork
- `byok_firecrawl_*` columns on `project_settings`, `firecrawl_cache` table, `_shared/firecrawl.ts` resolver. Domain allow-list + 24h cache + per-project page-cap + Langfuse tracing + audit log.
- New API: `POST /v1/admin/research/scrape`, `POST /v1/admin/research/search`, `GET /v1/admin/research/sessions[/:id]`.
- Settings page exposes Firecrawl key + allow-list + cap. No env fallback — Firecrawl is BYOK-only by design.

### Phase 2 — Apply web context where it pays
- **Research page** (`/research`) — admin-side scratchpad: search/scrape, save snippets, attach to a report or fix.
- **`fix-worker` augment** — when `judge_score < 0.6` OR RAG returns < 3 files, the worker proactively searches Firecrawl with the report symptoms and feeds the top hit into the LLM context. Audit-logged + traced.
- **`library-modernizer` cron** — weekly Edge Function inspects `package.json` of every connected repo, queries upstream release notes via Firecrawl, files `modernization_findings` rows. Admin "Library Modernization" card on `/intelligence` lists pending findings; "Dispatch fix" routes them through `fix-worker` as synthetic enhancement reports.

### Phase 3 — Schema & integration tightening
- **3a — BYOK NL→SQL & embeddings.** `_shared/nl-query.ts` and `_shared/embeddings.ts` now resolve Anthropic/OpenAI keys per-project via `resolveLlmKey`, with env fallback.
- **3b — Sentry Seer webhook.** `POST /v1/webhooks/sentry/seer` (HMAC-SHA256 verified) ingests Seer payloads as a complement to the polling cron. `_shared/seer.ts` consolidates the shared Sentry/Seer logic both paths now share.
- **3c — Device fingerprint anti-gaming.** `@mushi-mushi/core` ships `getDeviceFingerprintHash()` (SHA-256 of stable client characteristics). `reportSubmissionSchema` accepts it; `runCrossAccountCheck` flags devices that back ≥5 distinct reporter identities, surfaced in `/anti-gaming` under a new "Cross-account" badge.
- **3d — Graph NL queries.** `_shared/graph-nl.ts` short-circuits "blast radius / dependents / dependencies / path between" questions on `/query` to a recursive-CTE traversal of `graph_nodes`/`graph_edges`. Falls back to LLM SQL on detection miss; never hand-rolls graph CTEs in the LLM prompt anymore.
- **3e — `@mushi-mushi/plugin-sentry`.** New plugin (mirrors `plugin-pagerduty`): subscribes to `report.classified`, `fix.proposed`, `fix.applied`. Captures critical bugs into Sentry as events with stable fingerprint `mushi:<projectId>:<reportId>`, optionally resolves the matching Sentry issue when the fix lands. Registered in `plugin_registry`.

### Phase 4 — Prompt auto-tune (self-improving loop)
- `classification_evaluations.disagreement_reason` enum (`wrong_category`, `wrong_severity`, ..., `noise`, `other`) — judge now records *why* it disagreed, not just that it did.
- `prompt_versions` extended with `auto_generated`, `auto_generation_metadata`, `parent_version_id`.
- New `prompt-auto-tune` Edge Function (weekly cron, Sunday 07:00 UTC): pulls failing evaluations, buckets by `disagreement_reason`, asks the LLM for a revised prompt, files it as a `traffic_percentage = 0` candidate. Never auto-promotes — human approves via `/prompt-lab`.
- Prompt Lab UI gained an "Auto" badge + a Diff button that opens a side-by-side LCS diff against the parent prompt with the LLM's change-summary metadata.

### Phase 5 — Bounded PDCA orchestrator
- `scripts/pdca-bounded-loop.mjs` (`pnpm pdca:loop`): fires curated reports, polls triage + judge, dispatches fixes, polls dispatch status, writes `pdca-runs/<runId>/round-N.{jsonl,json}` and a roll-up `SUMMARY.md`. Stops at **3 clean rounds OR 5 attempts**. `PDCA_AUTOCOMMIT=true` makes it commit each round and the summary as it goes.

### Phase 6 — Advisor + lint cleanup
- `vector` extension moved out of `public` into `extensions` (`20260418005800_*`). `pg_net` left where it is — Supabase's build rejects `SET SCHEMA` (see Known issues #4).
- HIBP + MFA toggles documented (Auth → Providers → Email + Auth → MFA → enable WebAuthn / Passkeys).
- `packages/cli` exempts `no-console` (printing to stdout *is* the CLI's UI). `packages/mcp` test typed `any` → `unknown`. `pnpm -w lint` is now clean.

## Recovery cookbook

### Local admin keeps logging out
The cloud anon key in `apps/admin/src/lib/env.ts` should match what `supabase secrets list` returns. If it drifts, login throws "Invalid API key". Update both.

### Test user password drifts
```sql
update auth.users
   set encrypted_password = crypt('MushiTest2026!', gen_salt('bf')),
       updated_at = now()
 where email = 'test@mushimushi.dev';
```

### Pipeline silently stops processing reports
1. Check `processing_queue` — are reports `failed` with a `processing_error`?
2. If `Stage 1 failed: 401 UNAUTHORIZED_INVALID_JWT_FORMAT` → an internal function lost its `verify_jwt = false`. Redeploy with `--no-verify-jwt`.
3. If circuit breaker is open (every recent ingest = `status='queued'`), the recovery cron picks them up within 5 min; force it now via the `/queue` page → "Recover stranded" button, or `select * from public.recover_stranded_pipeline();` from SQL.
4. If you need a one-shot from the host: `node scripts/pipeline-recover.mjs` (requires `SUPABASE_URL` + `SUPABASE_ANON_KEY`).

### Langfuse traces vanish
1. Confirm `LANGFUSE_BASE_URL` matches your project's region (`us.cloud.langfuse.com`, NOT `cloud.langfuse.com`).
2. `_shared/observability.ts` accepts both `LANGFUSE_BASE_URL` and `LANGFUSE_HOST` as a fallback — but only one is needed.
3. `apps/admin/.env.local` must have `VITE_LANGFUSE_HOST` set or deep-links go to the wrong region.

### Sentry server-side traces vanish
`_shared/sentry.ts` reads `SENTRY_DSN_SERVER` first, `SENTRY_DSN` as fallback. Both are set on the cloud project.
