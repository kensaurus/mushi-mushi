### Why some functions have `verify_jwt = false`

`api`, `fast-filter`, `classify-report`, `judge-batch`, `fix-worker` all sit behind `verify_jwt = false` in `supabase/config.toml`. After Supabase rotated to the `sb_secret_*` key format, the platform gateway started 401-rejecting every service-role inter-function call (`UNAUTHORIZED_INVALID_JWT_FORMAT`). These functions are internal-only and self-validate via body params (`reportId` / `projectId`) plus `jwtAuth` middleware in `api`. **If you add a new internal function, add it to that block too.**

Cloud deploys must use `--no-verify-jwt`:

```bash
cd packages/server
npx supabase functions deploy fast-filter --no-verify-jwt
```

## Wave E ŌĆö Whitepaper gap closure + Firecrawl + self-improving loop (2026-04-18)

The "fill every Whitepaper V5 gap, add a self-improving loop, and bolt on optional web crawling" sweep. Six phases, all landed in this commit.

### Phase 1 ŌĆö Firecrawl BYOK groundwork
- `byok_firecrawl_*` columns on `project_settings`, `firecrawl_cache` table, `_shared/firecrawl.ts` resolver. Domain allow-list + 24h cache + per-project page-cap + Langfuse tracing + audit log.
- New API: `POST /v1/admin/research/scrape`, `POST /v1/admin/research/search`, `GET /v1/admin/research/sessions[/:id]`.
- Settings page exposes Firecrawl key + allow-list + cap. No env fallback ŌĆö Firecrawl is BYOK-only by design.

### Phase 2 ŌĆö Apply web context where it pays
- **Research page** (`/research`) ŌĆö admin-side scratchpad: search/scrape, save snippets, attach to a report or fix.
- **`fix-worker` augment** ŌĆö when `judge_score < 0.6` OR RAG returns < 3 files, the worker proactively searches Firecrawl with the report symptoms and feeds the top hit into the LLM context. Audit-logged + traced.
- **`library-modernizer` cron** ŌĆö weekly Edge Function inspects `package.json` of every connected repo, queries upstream release notes via Firecrawl, files `modernization_findings` rows. Admin "Library Modernization" card on `/intelligence` lists pending findings; "Dispatch fix" routes them through `fix-worker` as synthetic enhancement reports.

### Phase 3 ŌĆö Schema & integration tightening
- **3a ŌĆö BYOK NLŌåÆSQL & embeddings.** `_shared/nl-query.ts` and `_shared/embeddings.ts` now resolve Anthropic/OpenAI keys per-project via `resolveLlmKey`, with env fallback.
- **3b ŌĆö Sentry Seer webhook.** `POST /v1/webhooks/sentry/seer` (HMAC-SHA256 verified) ingests Seer payloads as a complement to the polling cron. `_shared/seer.ts` consolidates the shared Sentry/Seer logic both paths now share.
- **3c ŌĆö Device fingerprint anti-gaming.** `@mushi-mushi/core` ships `getDeviceFingerprintHash()` (SHA-256 of stable client characteristics). `reportSubmissionSchema` accepts it; `runCrossAccountCheck` flags devices that back Ōēź5 distinct reporter identities, surfaced in `/anti-gaming` under a new "Cross-account" badge.
- **3d ŌĆö Graph NL queries.** `_shared/graph-nl.ts` short-circuits "blast radius / dependents / dependencies / path between" questions on `/query` to a recursive-CTE traversal of `graph_nodes`/`graph_edges`. Falls back to LLM SQL on detection miss; never hand-rolls graph CTEs in the LLM prompt anymore.
- **3e ŌĆö `@mushi-mushi/plugin-sentry`.** New plugin (mirrors `plugin-pagerduty`): subscribes to `report.classified`, `fix.proposed`, `fix.applied`. Captures critical bugs into Sentry as events with stable fingerprint `mushi:<projectId>:<reportId>`, optionally resolves the matching Sentry issue when the fix lands. Registered in `plugin_registry`.

### Phase 4 ŌĆö Prompt auto-tune (self-improving loop)
- `classification_evaluations.disagreement_reason` enum (`wrong_category`, `wrong_severity`, ..., `noise`, `other`) ŌĆö judge now records *why* it disagreed, not just that it did.
- `prompt_versions` extended with `auto_generated`, `auto_generation_metadata`, `parent_version_id`.
- New `prompt-auto-tune` Edge Function (weekly cron, Sunday 07:00 UTC): pulls failing evaluations, buckets by `disagreement_reason`, asks the LLM for a revised prompt, files it as a `traffic_percentage = 0` candidate. Never auto-promotes ŌĆö human approves via `/prompt-lab`.
- Prompt Lab UI gained an "Auto" badge + a Diff button that opens a side-by-side LCS diff against the parent prompt with the LLM's change-summary metadata.

### Phase 5 ŌĆö Bounded PDCA orchestrator
- `scripts/pdca-bounded-loop.mjs` (`pnpm pdca:loop`): fires curated reports, polls triage + judge, dispatches fixes, polls dispatch status, writes `pdca-runs/<runId>/round-N.{jsonl,json}` and a roll-up `SUMMARY.md`. Stops at **3 clean rounds OR 5 attempts**. `PDCA_AUTOCOMMIT=true` makes it commit each round and the summary as it goes.

### Phase 6 ŌĆö Advisor + lint cleanup
- `vector` extension moved out of `public` into `extensions` (`20260418005800_*`). `pg_net` left where it is ŌĆö Supabase's build rejects `SET SCHEMA` (see Known issues #4).
- HIBP + MFA toggles documented (Auth ŌåÆ Providers ŌåÆ Email + Auth ŌåÆ MFA ŌåÆ enable WebAuthn / Passkeys).
- `packages/cli` exempts `no-console` (printing to stdout *is* the CLI's UI). `packages/mcp` test typed `any` ŌåÆ `unknown`. `pnpm -w lint` is now clean.

## CI / release model

- **`master` is protected.** No direct push. Every change goes via PR + the `Build & Test` required check.
- **Release workflow** (`.github/workflows/release.yml`) runs on push to master:
  - `Build & Verify` ŌĆö install / build / typecheck / lint / test (must pass)
  - `Version & Publish` ŌĆö Changesets action. If `.changeset/*.md` files exist ŌåÆ opens "Version Packages" PR. If that PR is merged ŌåÆ publishes to npm with provenance.
- **Adding a published change?** `pnpm changeset`, pick package(s), commit the resulting `.md`, push via PR.
- **Workflow itself** is supply-chain hardened (third-party actions pinned to commit SHAs).
