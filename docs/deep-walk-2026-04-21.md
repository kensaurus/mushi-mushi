## Confirmed real gaps

### 1. Manifest advertises endpoints that don't exist

`/v1/admin/auth/manifest` (lines 188-224 of
`packages/server/supabase/functions/api/index.ts`) advertises two endpoints
that have no Hono route:

```text
POST /v1/admin/auth/token            -> 404 (no app.post route)
POST /v1/admin/projects/:id/keys/rotate -> 404 (no app.post route)
```

Verified via direct curl against the dev API. Any external A2A client that
follows the manifest hits a 404 immediately. Wave S implements both; Wave T
adds a contract test so this can never regress silently.

### 2. No manifest contract test

There is no automated check that every endpoint advertised by the manifest
exists as a route. Without it, every manifest edit is a footgun.

### 3. No "dead button" lint

Empty `onClick={() => {}}` or `onClick={noop}` handlers are not caught by any
ESLint rule. Wave T adds one.

### 4. Skeleton + EmptyState + ResultChip parity

Skeletons and EmptyState components exist (`apps/admin/src/components/skeletons/*`,
30+ EmptyState consumers), but coverage across all 23 pages is uneven. Wave U
walks every page and tightens the parity.

## Wave U â€” list/table/visual polish (in progress)

Page-by-page sweep with two enforced standards:

1. **Btn `loading` prop instead of inline text swaps.** Audit found 25+ buttons
   that flipped a verb to its `-ing` form (`Save` â†’ `Savingâ€¦`, `Retry` â†’
   `Retryingâ€¦`, `Generate` â†’ `Generatingâ€¦`) while the `Btn` component already
   ships a built-in spinner via the `loading` prop. Inline swaps caused two
   problems: (a) buttons changed width mid-click, snapping the layout; (b) the
   spinner was never shown, so users couldn't tell a slow click from a dead one.
   All such buttons now use `loading={busy}` and keep the verb stable.

2. **Stagger-appear on freshly-loaded lists.** `useStaggeredAppear` was wired
   into one component (`InsightsRow`) despite shipping in Wave Q for repo-wide
   use. Wave U adds it to `FixesPage` (so a fresh batch of attempts cascades in
   instead of popping in one frame). `ReportsPage` already had inline stagger
   per-row (Wave M); left as-is to avoid a duplicate animation system.

3. **Sticky inline result on key actions.** Toast-only feedback fades after
   3-5s, leaving the user wondering whether their click landed. JudgePage's
   "Run judge now" + OnboardingPage's "Submit test report" + FirstReportHero
   now also paint a `<ResultChip>` next to the button, surviving the toast and
   recording when the action ran.
